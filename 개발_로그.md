## 2026-07-22 — Gemini 실제 활성화(키 연동) + 장소 검색 버그 수정

**AI 추천 실동작 확인**: 그동안 Gemini 코드는 있었지만 `GEMINI_API_KEY`가 없어 규칙 기반 폴백으로만 동작했다. 무료 티어 키를 발급받아 실제 연동을 완료했고, 추천 엔드포인트 end-to-end로 Gemini가 자연어 사유를 생성하는 것을 확인했다("기준 위치에서 매우 가까워 대중교통으로 편리하게 이동할 수 있습니다" 등).

이 과정에서 세 가지를 고쳤다:
- **`.env` 자동 로드 누락**: `python-dotenv`가 설치만 돼 있고 실제로 부르는 코드가 없어 `.env`를 만들어도 안 읽혔다. `wsgi.py`에서 `create_app` 임포트 전에 `load_dotenv()`를 호출하도록 추가(gunicorn `wsgi:app`으로 띄울 때도 모듈 로드 시점에 적용됨).
- **모델명**: 기본 모델 `gemini-2.0-flash`가 이 무료 티어 키에서 `limit: 0`(사용 불가)이라, 실제로 200을 반환하는 `gemini-flash-latest`로 기본값을 변경(`config.py`, `.env.example`, 어댑터 상수, 문서 전반).
- **호출 타임아웃 8초 → 20초**: 무료 티어 `gemini-flash-latest`의 구조화 출력 응답이 실측 5~8초라 8초로는 네트워크 변동 시 잦은 폴백(ReadTimeout)이 발생했다. 20초로 상향해 안정화.
- **점수 스케일**: 모델이 score를 0~100이 아닌 5점 척도(4.5 등)로 반환해 저장값이 0.045처럼 왜곡되던 것을, 프롬프트에 "0~100 정수, 별점 아님"을 명시해 교정.

**장소 검색 버그(사용자 리포트)**: 
- 지도 화면 내 검색(`/api/places/nearby`)이 현재 bbox 안에서만 조회돼 지도 밖 장소가 안 나오고, 이름 부분일치라 "경복궁"이 최상단에 안 오던 문제 → Nominatim 전역 지오코딩 결과를 앞에 병합하도록 수정.
- 사이드바 검색(`/api/places/search`)이 여행 지역을 검색어에 붙여("경복궁 부산") 질의해, 여행 지역이 서울이 아니면 **결과 0건**이 되던 심각한 버그 → region 이어붙이기를 제거하고 순수 검색어로 전역 질의.
- 관련 회귀 테스트 3건 추가, DTO→dict 변환 중복을 `_place_payload` 헬퍼로 통합.

**테스트**: 전 스위트 112 passed.

## 2026-07-21 — AI 추천: Claude API 연동 계획 → Gemini API로 전환

**배경**: 설계서·README에 "Claude API 연동 예정"으로 적혀 있었으나, 유료 API라 실제 연동 전 무료 대안을 검토했다. Google Gemini API(무료 티어), Groq API(무료 티어, 오픈모델), Ollama(로컬 self-host) 중 비교한 결과, 이 프로젝트가 이미 GPU 없는 Docker Compose 환경으로 배포되고(Ollama 배제), 추천 사유 생성이 지연시간에 민감하지 않아(Groq의 강점인 속도가 크게 중요하지 않음) **Gemini API**로 결정했다. 특히 Gemini의 `responseSchema` 구조화 출력 기능이 "이미 검증된 후보 풀에서만 골라야 한다"는 이 프로젝트의 추천 아키텍처(환각 차단 설계, 설계서 9.2.1절)와 잘 맞았다.

**변경 사항**:
- `app/adapters/gemini_adapter.py` 신규 — Gemini `generateContent` REST 호출, `responseSchema`로 `{place_id, score, reason}` 배열 강제, 실패 시 예외를 던져 상위 계층이 폴백하도록 함
- `app/services/recommend_service.py` — `_ai_adapter()`/`_score()` 헬퍼 추가. `GEMINI_API_KEY` 설정 시 Gemini를 우선 시도하고, 응답 검증(place_id가 후보 집합에 있는지) 후 사용. 미설정이거나 예외 발생 시 기존 규칙 기반 스코어링(`recommender.score_candidates`)으로 자동 폴백
- `app/engines/recommender.py` — 모듈 docstring에서 "Claude API 키가 없는 환경" 표현을 "Gemini 연동 여부는 recommend_service가 판단"으로 수정 (엔진 자체는 여전히 Flask/DB 비의존 순수 함수로 유지)
- `app/config.py`, `.env.example` — `GEMINI_API_KEY`, `GEMINI_MODEL`(기본 `gemini-2.0-flash`) 추가. `TestConfig`에는 `GEMINI_API_KEY = None`을 명시해 로컬 환경변수와 무관하게 테스트가 항상 규칙 기반 경로만 타도록 격리
- `tests/test_gemini_adapter.py` 신규 — 어댑터 단위 테스트(빈 후보 스킵, 정상 파싱, 비-리스트 응답 예외, HTTP 에러 예외)
- `tests/test_recommendations.py` — Gemini 성공 시 사유/점수가 그대로 반영되는지, 실패 시 규칙 기반 문구로 폴백되는지 통합 테스트 2건 추가
- `설계서.md` — 8.1/8.4/9.2절의 "Claude 연동 예정" 서술을 실제 구현(Gemini, 구조화 출력 방식, 폴백 조건)으로 갱신. 부록 B 참고자료도 Claude API 문서 → Gemini API 문서로 교체
- `README.md` — 핵심 기능/기술 스택에 Gemini 반영, 미구현 목록에서 "Claude API" 항목 제거하고 대신 아직 안 한 것(추천 결과 Redis 캐시, 피드백을 다음 추천에 실제 반영하는 로직)을 명시

**테스트**: `pytest -m "not e2e"` 109 → 115 passed (신규 6건 포함), 회귀 없음.

**아직 안 한 것**: 추천 결과의 Redis 캐시(무료 티어 쿼터 소진 방지), 수락/거절 피드백을 다음 추천 스코어링에 실제로 반영하는 로직(현재는 이력 저장까지만).
