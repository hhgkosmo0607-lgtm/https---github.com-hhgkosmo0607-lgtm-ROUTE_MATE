# RouteMate 사업계획서 (팀 공유용 개요)

> 이 문서는 설계서.md(상세 설계)·README.md(실행 가이드)·개발_로그.md(변경 이력)를 팀원이 처음부터 다 읽지 않아도 프로젝트의 전체 그림과 현재 상태를 파악할 수 있도록 요약한 자료다. 세부 스펙은 각 원본 문서를 참고한다.

---

## 1. 한 줄 요약

**"장소 선택부터 경로 최적화, 돌발 상황 대응(Plan B)까지 — 여행 계획의 전 과정을 하나의 흐름으로 해결하는 지도 기반 여행 설계 플랫폼."**

기존 여행 도구는 "장소 나열"에 그치지만, RouteMate는 경로 최적화 엔진을 중심에 두고 이동시간·거리·사용자 취향을 반영해 실제로 실행 가능한 일정을 자동 생성한다.

---

## 2. 문제 정의 (왜 만드는가)

- **도구 분절**: 장소 검색(포털·SNS) · 동선 확인(지도 앱) · 일정 정리(메모·스프레드시트) · 예산 관리(별도 앱)가 흩어져 있어 계획 수립에 시간이 과도하게 든다.
- **비효율적 동선**: 지리 정보를 고려하지 않고 나열식으로 장소를 배치하면 왕복 이동이 반복돼 이동 시간이 관광 시간을 잠식한다.
- **돌발 상황 대응 부재**: 기존 서비스는 계획 "수립"에만 집중하고, 여행 "실행 중" 발생하는 웨이팅·휴무·우천 같은 변수에 대한 대응 수단이 없다.
- **획일적 추천**: 대부분의 추천이 인기순 나열에 그쳐 알레르기·이동수단·예산 같은 개인 제약 조건이 반영되지 않는다.

## 3. 솔루션 — 핵심 기능

| 기능 | 설명 | 상태 |
| --- | --- | --- |
| 경로 자동 최적화 | Google OR-Tools(VRPTW)로 Day 배치·방문 순서 동시 최적화, 솔버 장애 시 휴리스틱 자동 폴백 | ✅ 완료 |
| 지도 기반 설계 | Leaflet+OpenStreetMap. 검색·지도·일정 3단 화면, 주변 POI 탐색 | ✅ 완료 |
| 실경로 이동시간 | OSRM 실도로 거리행렬, 실패 시 근사치 폴백 | ✅ 완료 |
| 일정 편집 | 순서/Day 이동, 체류시간 수정, 부분 재계산 | ✅ 완료 |
| 맞춤 추천 | 하드 필터(알레르기·예산) + Gemini API(자연어 사유 생성) — 미설정/실패 시 규칙 기반 폴백 | 🟡 부분 완료 |
| Plan B | 대체 장소 등록 → 발동 → 재구성 미리보기 → 승인/거절 → 되돌리기 | 🟡 수동 발동만 (자동 감지 없음) |
| 부가 기능 | 체크리스트, 예산 관리, 공유 링크, 동행자 초대, 여행 복제 | ✅ 완료 |
| 보안 | bcrypt, CSRF, CSP, 레이트리밋, 알레르기 정보 AES-256-GCM 암호화 | ✅ 완료 |

## 4. 시장 포지셔닝 (경쟁 비교 요약)

| 비교 항목 | 일정 플래너형 서비스 | 지도 저장형 서비스 | AI 추천형 서비스 | **RouteMate** |
| --- | --- | --- | --- | --- |
| 자동 경로 최적화 | 미지원(수동 배치) | 미지원 | 부분 지원 | **핵심 기능** |
| Day 자동 배치 | 수동 | 미지원 | 부분 지원 | **자동** |
| 돌발 상황 대응(Plan B) | 미지원 | 미지원 | 미지원 | **핵심 차별점** |
| AI 개인화 추천 | 미지원 | 미지원 | 지원(범용) | 프로필 기반(제약 조건 반영) |

**시사점**: 경로 "최적화"와 여행 중 "재구성"은 경쟁 서비스가 공통으로 비어 있는 영역 — RouteMate의 포지셔닝 축. AI 추천 자체는 이미 보편화됐으므로 "제약 조건 준수 + 추천 사유 제시"로 차별화한다.

*(출처: 설계서.md 2장 — 서비스명은 일반화 표기)*

## 5. 기술 스택

| 구분 | 기술 |
| --- | --- |
| Backend | Python 3.12 / Flask 3 · SQLAlchemy 2 · SQLite(개발) → MySQL(운영) |
| 경로 최적화 | Google OR-Tools (VRPTW), 폴백: NN+2-opt |
| 지도/데이터 | Leaflet + OSM · Nominatim(검색) · Overpass(POI) · OSRM(경로) — 전부 무료, 공개 데모 서버 |
| AI 추천 | Google Gemini API (`gemini-2.0-flash`, 무료 티어) — 미설정 시 규칙 기반 폴백 |
| 배포 | Docker Compose(web/db/redis) · Gunicorn · GitHub Actions CI |

## 6. 지금 어디까지 됐나 (진행 상황)

### 완료된 것
- 회원가입/로그인/보안 전반, 여행·일정 CRUD, 경로 자동 생성(OR-Tools), 지도 검색/POI, Plan B 수동 흐름, 체크리스트/예산/공유/동행자, 단위+통합 테스트 109개 + E2E 5종
- AI 추천: Gemini API 연동 완료(2026-07-21) — 무료 티어, 실패 시 규칙 기반으로 자동 폴백하는 구조

### 아직 안 된 것 (팀에서 우선순위 논의 필요)
| 항목 | 현재 상태 | 비고 |
| --- | --- | --- |
| 기상청(날씨) 연동 | 미착수 | Plan B 우천 트리거가 **수동 발동만** 가능 |
| 카카오 OAuth | 미착수 | `place.ext_source`에 컬럼값만 예약돼 있고 실제 로그인 연동 없음 |
| TourAPI | 미착수 | 추천 후보가 서비스 DB(OSM 수집분)로만 한정 |
| SMTP(실메일 발송) | 코드는 있으나 미설정 | 비밀번호 재설정 메일이 지금은 서버 로그로만 출력(개발 모드) |
| AI 추천 결과 캐시 | 미착수 | Gemini 무료 티어 쿼터 소진 방지용 Redis 캐시 필요 |
| 추천 피드백 학습 | 미착수 | 수락/거절 이력은 저장되지만 다음 추천에 반영 안 됨 |
| 운영 인프라 확장 | 미착수 | 지도 API가 무료 공개 데모 서버 — 트래픽 늘면 자체 인스턴스 전환 필요, Redis/Celery도 아직 미도입 |

### 이번 주 발견한 운영상 문제
- **로컬 개발 환경 셋업이 취약함**: `flask db upgrade`(DB 마이그레이션)를 빠뜨리면 로그인을 포함한 모든 DB 관련 API가 500 에러(HTML 디버그 페이지)를 뱉는다. 프론트가 이걸 JSON으로 파싱하지 못해 "서버 응답을 처리할 수 없음"처럼 보이는 증상으로 나타남 — **신규 팀원 온보딩 시 반복될 가능성이 높은 실수**. 초기화를 자동으로 검증/안내하는 스크립트가 없다.
- **테스트 계정 미시딩**: README에 `test@test.com`/`test1234`가 안내돼 있지만 이를 자동 생성하는 시드 스크립트가 없어, DB를 새로 만들 때마다 수동 회원가입이 필요하다.
- **가상환경 격리 필요**: 프로젝트 의존성(`ortools` 등)을 팀 공용/다른 프로젝트와 같이 쓰는 Python 환경에 설치하면 numpy/protobuf 버전 충돌이 발생할 수 있다 — 반드시 프로젝트 전용 가상환경(`.venv`)을 사용해야 한다.

| 부하 대응 | 동시 사용자 100명, 조회 p95 500ms 목표 | 30명 통과 / 100명은 p95 1.4s로 목표 초과 — 확장 필요 |

## 8. 제안하는 다음 우선순위

팀 논의를 위한 초안 제안 (합의된 순서는 아님):

1. **온보딩 자동화** — DB 마이그레이션 자동 실행 + 테스트 계정 시드 스크립트 (신규 팀원 셋업 실패 방지, 비용 낮고 효과 큼)
2. **기상청 API 연동** — Plan B 자동 발동은 서비스의 핵심 차별점인데 지금은 수동뿐이라 체감 효과가 약함
3. **AI 추천 캐시** — Gemini 무료 티어 쿼터 소진 리스크 방지
4. **카카오 OAuth / SMTP 실연동** — 실사용자 확보 단계에서 필요
