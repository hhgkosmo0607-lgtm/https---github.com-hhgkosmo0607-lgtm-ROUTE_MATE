## 1. 프로젝트 개요

### 1.1 프로젝트 소개

RouteMate는 사용자가 가고 싶은 장소를 직접 선택하거나 AI 추천을 받아 효율적인 여행 경로를 설계하는 통합 여행 계획 웹서비스이다. 기존 여행 계획 도구가 "장소를 나열"하는 수준에 머무르는 것과 달리, RouteMate는 경로 최적화 엔진(Route Engine)을 중심에 두고 이동시간·거리·영업시간·사용자 취향을 종합적으로 반영한 실행 가능한 일정을 자동으로 생성한다.

또한 여행 중 흔히 발생하는 돌발 상황(맛집 웨이팅, 예상치 못한 휴무, 우천 등)에 대비해 대체 일정을 사전에 등록하고, 상황 발생 시 잔여 일정을 자동으로 재구성하는 Plan B 기능을 핵심 차별점으로 제공한다.

|  |
| --- |
| **한 줄 정의**  "장소 선택부터 경로 최적화, 돌발 상황 대응까지 — 여행 계획의 전 과정을 하나의 흐름으로 해결하는 지도 기반 여행 설계 플랫폼" |

### 1.2 개발 배경 및 필요성

#### 1.2.1 문제 인식

* 여행 계획 과정의 분절 — 장소 검색(포털·SNS), 동선 확인(지도 앱), 일정 정리(메모·스프레드시트), 예산 관리(별도 앱)가 서로 다른 도구에 흩어져 있어 계획 수립에 과도한 시간이 소요된다.
* 비효율적 동선 — 지리 정보를 고려하지 않고 장소를 나열식으로 배치하면 왕복 이동이 반복되어 실제 여행에서 이동 시간이 관광 시간을 잠식한다.
* 돌발 상황 대응 부재 — 기존 서비스는 계획 "수립"에만 집중하고 여행 "실행 중" 발생하는 웨이팅·휴무·우천 등의 변수에 대한 대응 수단을 제공하지 않는다.
* 획일적 추천 — 대부분의 추천은 인기순 나열에 그쳐 알레르기, 이동수단, 걷기 선호도, 예산 같은 개인 제약 조건이 반영되지 않는다.

#### 1.2.2 해결 방향

* 검색·추천·경로·일정·예산·공유를 단일 서비스로 통합하여 도구 전환 비용을 제거한다.
* 거리 행렬 기반 경로 최적화 알고리즘으로 총 이동시간을 최소화한 Day별 일정을 자동 생성한다.
* Plan B 엔진을 통해 조건(웨이팅·휴무·우천) 감지 → 대체 장소 투입 → 잔여 일정 재계산의 자동화된 대응 체계를 구축한다.
* 여행 프로필(스타일·음식 취향·알레르기·이동수단·예산 등)을 구조화하여 AI 추천의 입력 컨텍스트로 활용, 실질적인 개인화를 구현한다.

### 1.4 프로젝트 범위

#### 1.4.1 포함 범위 (In Scope)

* 회원 관리: 회원가입, 로그인/로그아웃, 여행 프로필 설정 및 수정
* 직접 여행 설계: 장소 검색·추가, 자동 경로 생성, Day 자동 배치, 이동시간·거리 계산, 드래그 앤 드롭 순서 수정 및 자동 재계산
* AI 여행 설계: 여행지·맛집·카페 추천, 빈 일정 슬롯 추천, 사용자 취향 기반 개인화 추천
* Plan B: 대체 장소 등록, 웨이팅·휴무·우천 조건 대응, 일정 자동 재구성
* 지도: 경로 폴리라인 표시, 지도 위 장소 선택, 구간별 이동시간·거리 확인
* 부가 기능: 체크리스트, 예산 관리, 일정별 메모, 여행 계획 공유(링크·동반자 초대)

#### 1.4.2 제외 범위 (Out of Scope)

* 숙박·교통편 실시간 예약 및 결제 기능 (외부 링크 연결로 대체)
* 네이티브 모바일 앱 (반응형 웹으로 모바일 대응)
* 해외 여행지 지원 (1차 릴리스는 국내 여행 대상, 확장성만 확보)
* 실시간 채팅 등 동반자 간 커뮤니케이션 기능

### 1.6 용어 정의

|  |  |
| --- | --- |
| **용어** | **정의** |
| 여행(Trip) | 제목·기간·지역을 갖는 하나의 여행 계획 단위. 여러 Day와 일정을 포함한다. |
| 일정(Schedule) | 특정 Day의 특정 순서에 배치된 장소 방문 단위. 시작 시간, 이동시간·거리를 갖는다. |
| 장소(Place) | 지도 좌표·카테고리·영업정보를 갖는 방문 대상. 지도/관광 API에서 수집한다. |
| Route Engine | 장소 목록을 입력받아 좌표 변환 → 거리/시간 행렬 → 경로 최적화 → Day 배치 → 일정 생성을 수행하는 핵심 모듈. |
| Plan B | 특정 일정에 연결된 대체 장소와 발동 조건(웨이팅·휴무·우천·수동)의 집합. |
| 발동 조건(Trigger) | Plan B가 활성화되는 상황 유형. WAIT(웨이팅), CLOSED(휴무), RAIN(우천), MANUAL(수동)로 구분한다. |
| 여행 프로필(Profile) | 여행 스타일·음식 취향·알레르기·이동수단·걷기 선호도·예산·관심사(카페/사진/쇼핑/휴식)를 구조화한 사용자 취향 데이터. |
| 거리 행렬(Distance Matrix) | N개 장소 간 모든 쌍의 이동거리·이동시간을 담은 N×N 행렬. |
| 2-opt | 경로에서 두 간선을 교차 재연결하여 총 거리를 줄이는 경로 개선 휴리스틱 알고리즘. |
| NN(Nearest Neighbor) | 현재 위치에서 가장 가까운 미방문 장소를 다음 목적지로 선택하는 초기 경로 생성 휴리스틱. |
| 빈 일정 추천 | 일정 사이의 시간 공백을 감지하여 주변 장소를 채워 넣는 AI 추천 기능. |
| 하드 필터 | 알레르기·예산 초과 등 절대 위반하면 안 되는 제약을 추천 후보에서 사전 제거하는 필터. |

## 2. 시장 분석 및 벤치마킹

### 2.1 시장 동향

코로나19 이후 국내 여행 수요가 회복되면서 개인화된 여행 계획 도구에 대한 관심이 높아지고 있다. 특히 MZ세대를 중심으로 "계획형 여행"과 "맛집 중심 여행" 문화가 확산되며, 웨이팅·휴무 같은 변수에 대한 사전 대비 수요가 뚜렷하게 나타난다. 동시에 생성형 AI의 확산으로 자연어 기반 여행 추천에 대한 사용자 기대치가 크게 상승했다.

* 여행 계획에 평균 3개 이상의 앱/서비스를 병행 사용하는 것으로 조사됨 (검색·지도·메모 분절)
* AI 추천을 경험한 사용자의 상당수가 "추천 이유가 불투명하다", "내 제약 조건(알레르기 등)을 모른다"는 불만을 보임
* 일정 관리 도구는 "만들 때"만 쓰이고 "여행 중"에는 지도 앱으로 이탈하는 사용 패턴이 일반적

### 2.3 시사점

* 경로 "최적화"와 여행 중 "재구성"은 경쟁 서비스가 공통적으로 비어 있는 영역으로, RouteMate의 포지셔닝 축으로 삼는다.
* AI 추천은 이미 보편화 단계이므로, 추천 자체보다 "제약 조건 준수 + 추천 사유 제시 + 원클릭 일정 반영"으로 차별화한다.
* 여행 중 사용을 위해 지도 화면과 Plan B 발동 UI를 모바일 반응형 최우선 순위로 설계한다.

### 3.4 제약 사항

* 외부 API 무료 쿼터 한도(지도 API 일일 호출량, TourAPI 일 1,000건 등) 내에서 동작하도록 캐싱·배치 조회를 설계에 반영한다.
* AI API 호출 비용을 고려하여 추천 1회당 호출 1건 이하, 후보 필터링은 서버에서 선처리한다.
* 개발 기간(1학기)과 팀 규모(4인)를 고려하여 Must 요구사항을 우선 구현하고 Could 항목은 시연 이후 확장한다.
* 개인정보보호법에 따라 수집 항목을 최소화(이메일·닉네임·취향 정보)하고 이용 목적을 약관에 명시한다.

## 4. 시스템 아키텍처

### 4.1 시스템 구성도

RouteMate는 3계층(Client — Flask Application Server — Database) 구조를 기본으로 하며, Flask 서버가 4종의 외부 API(지도·관광·날씨·AI)와 연동한다. 핵심 비즈니스 로직인 Route Engine, AI Recommendation, Plan B Engine은 컨트롤러와 분리된 독립 모듈로 구성하여 테스트 용이성과 재사용성을 확보한다.

### 4.2 데이터 흐름 요약

|  |  |  |
| --- | --- | --- |
| **흐름** | **경로** | **설명** |
| 화면 요청 | Browser → Flask → Browser | Jinja2 템플릿 렌더링. 인증이 필요한 화면은 세션 검증 후 응답 |
| 일정 데이터 | Browser ↔ Flask ↔ MySQL | REST API(JSON)로 여행·일정 CRUD, ORM을 통한 트랜잭션 처리 |

| 좌표·경로 조회 | Flask ↔ 지도 API | 지오코딩, 구간별 이동시간·거리 조회. 결과는 Redis 캐싱 |
| 장소 후보 수집 | Flask ↔ 관광 API | 지역·카테고리별 관광지·음식점 정보 수집(추천 후보) |

## 6. 데이터베이스 설계

### 6.1 설계 원칙

* 제3정규형(3NF)을 기준으로 정규화하되, 일정 목록 조회 성능을 위해 SCHEDULE에 이동시간·거리(move_min, move_km)를 계산 결과로 반정규화 저장한다.
* 모든 테이블은 대리키(AUTO_INCREMENT BIGINT)를 PK로 사용하고, 비즈니스 식별자(이메일, 공유 토큰)는 UNIQUE 제약으로 분리한다.
* 삭제는 소프트 삭제(deleted_at)를 기본으로 하여 공유된 여행의 참조 무결성과 복구 가능성을 확보한다.
* 외래키에는 ON DELETE 정책을 명시한다 — 여행 삭제 시 하위 일정·체크리스트 등은 CASCADE, 장소 마스터(PLACE)는 RESTRICT.
* 문자 집합은 utf8mb4, 시각 컬럼은 DATETIME(서버 UTC 저장, 표시 시 KST 변환)을 사용한다.

### 6.7 데이터 수명 주기

* PLACE 캐시 데이터는 cached_at 기준 24시간 경과 시 재조회로 갱신하며, 영업정보 변경(휴무 등)을 반영한다.
* 탈퇴 회원(deleted_at 설정)의 개인정보(email, nickname)는 익명화 배치로 30일 내 파기한다. 공유 여행의 일정 데이터는 "탈퇴한 사용자" 표기로 보존한다.
* RECOMMENDATION 이력은 1년 보관 후 통계 요약만 남기고 삭제한다.

## 7. API 설계

### 7.1 설계 규칙

* REST 원칙을 따르며 자원은 소문자 복수형 명사(/api/trips), 행위는 HTTP 메서드로 표현한다.
* 요청·응답 본문은 JSON(UTF-8)이며, 모든 응답은 공통 래퍼 { "success": bool, "data": ..., "error": ... } 형식을 따른다.
* 인증은 세션 쿠키 기반으로 하며, 인증 필요 API는 미인증 시 401, 권한 부족 시 403을 반환한다.
* 목록 API는 페이지네이션(?page=1&size=20)을 지원하고 최대 size는 50으로 제한한다.
* 변경 API(POST/PUT/DELETE)는 CSRF 토큰 헤더(X-CSRF-Token)를 요구한다(NFR-03).

## 9. 핵심 알고리즘 설계

본 장에서는 RouteMate의 3대 핵심 엔진(Route Engine, AI Recommendation Engine, Plan B Engine)의 내부 동작을 의사코드 수준으로 상세히 기술한다. 세 엔진은 모두 순수 Python 모듈(services/ 계층)로 구현되어 Flask 라우트 계층과 분리되며, 단위 테스트가 가능하도록 외부 API 의존성은 어댑터 인터페이스로 주입받는다.

### 9.1 Route Engine — 최적 경로 설계

#### 9.1.1 문제 정의

사용자가 선택한 n개의 장소를 "총 이동 시간이 최소가 되는 방문 순서"로 정렬하고, 각 장소의 체류 시간과 하루 활동 가능 시간을 고려하여 Day 단위로 배치하는 문제다. 이는 외판원 문제(TSP)의 변형으로 NP-hard에 해당하므로, 여행 계획에서 실용적인 규모(장소 5~30개)에 대해 근사 최적해를 1초 이내(NFR-06)에 산출하는 휴리스틱 접근을 채택한다.

|  |
| --- |
| **설계 결정 (v1.1 개정)**  완전 탐색(O(n!))이나 동적 계획법(O(n²·2ⁿ))은 n=20 이상에서 응답 시간 요구를 만족할 수 없다. 초기 설계(v1.0)는 최근접 이웃(NN)+2-opt 휴리스틱이었으나, "전체 순회를 하나 풀고 → 탐욕적으로 Day를 분할"하는 2단계 구조는 Day 경계에서 동선이 어긋나는 근본 한계가 있다. v1.1은 문제를 **VRPTW(시간창 있는 차량 경로 문제)**로 재정의하고 산업용 솔버 **Google OR-Tools**(GUIDED_LOCAL_SEARCH)로 Day 배치와 방문 순서를 **동시에** 최적화한다. Day를 차량, 일일 활동시간을 시간창, 배치 불가 장소를 페널티 드롭(→미배치 보관함)으로 모델링한다. 실측 기준 NN+2-opt 대비 총 이동시간 3~27%(평균 약 18%) 추가 단축. NN+2-opt는 솔버 실패·미설치 시의 폴백으로 유지한다(8.3절 폴백 원칙). |

#### 9.1.2 처리 단계

|  |  |  |  |  |
| --- | --- | --- | --- | --- |
| **단계** | **이름** | **입력** | **출력** | **설명** |
| 1 | 좌표 확정 | 장소 목록 | 좌표 목록 | PLACE 테이블의 위경도 사용, 누락 시 지도 API 지오코딩 |
| 2 | 거리 행렬 구축 | 좌표 목록, 이동수단 | n×n 시간 행렬 | 지도 API 길찾기 호출(캐시 우선), 실패 시 Haversine 근사 |
| 3 | VRPTW 최적화 | 시간 행렬, 여행 일수, 활동시간 | Day별 방문 순서 | OR-Tools 솔버 — Day 배치·순서 동시 최적화, 배치 불가분은 드롭 |
| 3′ | (폴백) NN+2-opt | 시간 행렬, 시작점 | 방문 순서 → Day 분할 | 솔버 실패·미설치 시: 최근접 이웃 초기해 + 2-opt 개선 + 탐욕 Day 분할 |
| 4 | 일정 생성 | Day별 일정 | SCHEDULE 레코드 | 시작 시각부터 순차 배치, 이동시간·거리 부여 |

#### 9.1.3 의사코드

```python
def build_route(places, transport, days, day_start, day_end, profile):
    coords = ensure_coords(places)                     # 1단계
    M = time_matrix(coords, transport)                 # 2단계, 캐시 우선
    # 3단계: Nearest Neighbor 초기해
    order = [start_index(places)]
    remaining = set(range(len(places))) - set(order)
    while remaining:
        cur = order[-1]
        nxt = argmin(remaining, key=lambda j: M[cur][j])
        order.append(nxt); remaining.remove(nxt)
    # 4단계: 2-opt 개선 (개선 없을 때까지 반복, 상한 200회)
    improved = True
    while improved:
        improved = False
        for i in range(1, len(order) - 2):
            for k in range(i + 1, len(order) - 1):
                if gain(order, i, k, M) > EPSILON:
                    order[i:k+1] = reversed(order[i:k+1])
                    improved = True
    # 5~6단계: Day 배치 + 시각 부여
    schedule = assign_days(order, M, places, days,
                           day_start, day_end,
                           walk_limit=profile.walk_preference)
    return schedule
```

#### 9.1.4 Day 배치 규칙

* 일일 활동 시간 = day_end − day_start(기본 09:00~21:00, 여행 생성 시 사용자 지정 가능).
* 누적 시간(체류 + 이동)이 일일 활동 시간을 초과하기 직전 장소에서 해당 Day를 마감하고, 다음 장소부터 새 Day를 시작한다.
* 11:00~13:30, 17:30~19:30 구간에 카테고리=음식점 장소가 오도록 인접 순서 내에서 국소 스왑(swap)을 시도한다. 스왑으로 총 이동 시간이 10% 이상 증가하면 스왑을 취소한다.
* 프로필의 걷기 선호도가 "적게"인 경우 도보 연속 이동 25분 초과 구간에 휴식(카페) 삽입을 제안 플래그로 표시한다.
* Day 수가 여행 일수를 초과하면 초과 장소를 "미배치 보관함"으로 이동시키고 사용자에게 알린다(FR-207).

#### 9.1.5 부분 재계산(드래그 수정 대응)

사용자가 일정을 드래그로 수정하면(FR-209) 전체 경로를 다시 풀지 않고, 변경된 Day의 이동 시간만 거리 행렬에서 조회하여 O(d) 시간(d=해당 Day의 장소 수)에 갱신한다. 거리 행렬은 여행 편집 세션 동안 Redis에 보관(TTL 30분)하므로 추가 외부 API 호출이 발생하지 않는다. 사용자가 명시적으로 "다시 최적화" 버튼을 누른 경우에만 9.1.3의 전체 파이프라인을 재실행한다.

#### 9.1.6 복잡도 및 성능 분석

|  |  |  |
| --- | --- | --- |
| **항목** | **복잡도/수치** | **비고** |
| 거리 행렬 구축 | O(n²) API 호출 | 캐시 적중 시 0회, 배치 요청으로 왕복 최소화 |

### 9.2 AI Recommendation Engine — 취향 기반 추천

#### 9.2.1 파이프라인 개요

추천 엔진은 "후보 수집 → 하드 필터 → LLM 스코어링/사유 생성 → 후처리" 4단계 파이프라인이다. LLM은 후보를 생성하는 역할이 아니라 이미 검증된 후보 풀(관광 API + 지도 API 실데이터)을 사용자 프로필에 맞게 선별·설명하는 역할만 수행한다. 이 구조는 존재하지 않는 장소를 추천하는 환각(hallucination) 문제를 원천 차단한다.

|  |  |  |
| --- | --- | --- |
| **단계** | **처리 내용** | **데이터 원천** |
| **① 후보 수집** | 여행 지역 반경 내 카테고리별 장소 30~50개 수집 | 관광 API, 지도 API(캐시 우선) |
| **② 하드 필터** | 알레르기 위험 업종 제외, 휴무일 제외, 이미 일정에 포함된 장소 제외, 예산 상한 초과 제외 | PROFILE, SCHEDULE, 영업정보 |
| **③ LLM 스코어링** | 프로필+여행 맥락을 프롬프트로 전달, 후보별 적합도 점수(0~100)와 추천 사유(1문장) JSON 생성 | AI API(Claude) |
| **④ 후처리** | 점수순 정렬, 상위 5개 선정, 현재 일정과의 거리 페널티 반영, RECOMMENDATION 테이블 기록 | Route Engine 거리 행렬 |

#### 9.2.2 프롬프트 설계

LLM 호출은 시스템 프롬프트에 출력 스키마를 고정하고, 사용자 프롬프트에 프로필·맥락·후보 목록을 구조화하여 전달한다. 응답은 JSON만 허용하며 파싱 실패 시 1회 재시도 후 규칙 기반 폴백으로 전환한다(8.3절).

```text
[System]
너는 여행 추천 어시스턴트다. 반드시 아래 JSON 배열만 출력한다.
[{"place_id": int, "score": 0-100, "reason": "20자 내외 한 문장"}]
후보 목록에 없는 장소를 만들어내지 마라.
[User]
## 사용자 프로필
- 여행 스타일: 맛집 중심 / 널널한 일정 선호
- 음식 취향: 해산물 선호, 매운 음식 비선호 / 알레르기: 갑각류
- 이동: 대중교통, 걷기 적게 / 1일 예산: 10만원
- 선호 활동: 카페 ○, 사진 ○, 쇼핑 ×, 휴식 ○
## 여행 맥락
- 지역: 부산 / Day 2 오후 15:00~18:00 빈 일정
- 직전 장소: 감천문화마을 / 다음 장소: 자갈치시장(18:30)
## 후보 목록 (JSON)
[{"place_id":41,"name":"흰여울문화마을","category":"관광지", ...}, ...]
```

#### 9.2.3 스코어링 보정

LLM 점수(0~100)에 위치 적합성을 결합한 최종 점수로 정렬한다.

```python
final = 0.7 * llm_score
      + 0.3 * (100 - min(100, detour_minutes * 4))
# detour_minutes: 직전→후보→다음 경유 시 추가되는 이동 시간(분)
```

* 동점 시 관광 API 인기 순위가 높은 장소를 우선한다.
* 추천 결과는 RECOMMENDATION 테이블에 프롬프트 요약·점수와 함께 저장하여, 수락/거절 피드백을 다음 추천의 프로필 보정에 활용한다(FR-306).

#### 9.2.4 비용·지연 관리

* 후보 목록은 이름·카테고리·태그·가격대 등 최소 필드만 전달하여 프롬프트를 2,000토큰 이내로 유지한다.
* 동일 (지역, 시간대, 프로필 해시) 조합의 추천 결과는 Redis에 1시간 캐시한다.
* AI API 지연 목표는 p95 4초이며, 8초 타임아웃 초과 시 규칙 기반 폴백 결과를 즉시 반환한다.

### 9.3 Plan B Engine — 상황 대응 일정 재구성

#### 9.3.1 발동 조건

|  |  |  |  |
| --- | --- | --- | --- |
| **조건 코드** | **감지 방법** | **감지 시점** | **기본 동작** |
| PB-RAIN(우천) | 날씨 API 예보에서 해당 Day 강수확률 ≥ 60% | 매일 06:00 배치 + 일정 조회 시 | 실외 장소에 대체 제안 알림 |
| PB-CLOSED(휴무) | 관광/지도 API 영업 정보와 일정 시각 대조 | 일정 저장 시 + 매일 06:00 배치 | 해당 장소에 경고 배지 + 대체 제안 |
| PB-WAIT(웨이팅) | 사용자가 현장에서 수동 발동 | 실시간(사용자 조작) | 즉시 대체 장소 목록 표시 |
| PB-MANUAL(기타) | 사용자가 임의 장소에 수동 발동 | 실시간(사용자 조작) | 즉시 대체 장소 목록 표시 |

#### 9.3.2 대체 장소 선정 규칙

* 사용자가 사전 등록한 Plan B 장소(PLAN_B 테이블, FR-401)가 있으면 최우선으로 제시한다.
* 사전 등록이 없으면 동일 카테고리 + 반경 2km 내 + 현재 영업 중 조건으로 후보를 검색한다.
* 우천 조건(PB-RAIN)에서는 실내 태그(실내 관광지, 박물관, 카페, 쇼핑몰)를 가진 장소만 후보로 한다.
* 후보에 9.2절 추천 엔진의 하드 필터(알레르기·예산·중복)를 동일하게 적용한다.
* 점수 = 카테고리 일치(40) + 근접도(30) + 프로필 적합도(30)로 상위 3개를 제시한다.

#### 9.3.3 일정 재구성 의사코드

```python
def activate_plan_b(schedule_id, alt_place, reason):
    sch  = load_schedule(schedule_id)
    day  = sch.day_items()                  # 해당 Day의 일정 목록
    old  = sch.place
    replace(sch, alt_place)                 # ① 장소 교체(원본은 이력 보존)
    # ② 해당 Day만 이동시간 재계산 (부분 재계산, 9.1.5 재사용)
    recalc_times(day, matrix_for(day))
    # ③ 시간 충돌 해소: 뒤 일정을 순차적으로 밀되
    #    day_end 초과분은 미배치 보관함으로 이동 + 사용자 확인 요청
    overflow = shift_and_collect_overflow(day, day_end)
    if overflow:
        notify_user(overflow, action="confirm_or_park")
    log_planb_history(sch, old, alt_place, reason)   # ④ 이력 기록
    return day
```

* 교체 이력은 PLAN_B.activated_at과 SCHEDULE의 원본 참조로 보존되어, 사용자는 "되돌리기"로 원래 일정을 복원할 수 있다.
* 재구성은 해당 Day에 국한되며 다른 Day는 변경하지 않는다. 전체 재최적화가 필요하면 사용자가 명시적으로 요청해야 한다(예측 가능성 원칙).

#### 9.3.4 알림 정책

* 자동 감지(PB-RAIN, PB-CLOSED)는 앱 내 알림 배지와 여행 설계 화면 상단 배너로만 안내하며, 사용자의 확인 없이 일정을 자동 변경하지 않는다.
* 동일 조건의 중복 알림은 24시간 내 1회로 제한한다.

## 11. 보안 설계

### 11.1 인증·인가

* 세션 기반 인증(Flask-Login)을 사용하며, 세션 쿠키는 HttpOnly + Secure + SameSite=Lax로 설정한다.
* 비밀번호는 bcrypt(cost 12)로 해시하여 저장하고, 평문·복호화 가능 방식은 사용하지 않는다(NFR-01).
* 로그인 5회 연속 실패 시 10분 계정 잠금, 잠금 이력은 감사 로그에 기록한다(NFR-02).
* 인가는 리소스 소유 검사 데코레이터(@require_trip_member)로 일원화한다. 모든 /api/trips/{id}/* 요청은 TRIP_MEMBER 조회로 접근 권한을 검증하며, 타인 리소스 접근 시 404를 반환해 존재 여부를 숨긴다.

### 11.2 입력 검증과 주요 취약점 대응

|  |  |  |
| --- | --- | --- |
| **위협** | **대응 방안** | **적용 위치** |
| SQL Injection | SQLAlchemy ORM 파라미터 바인딩만 사용, 원시 쿼리 금지 | DAO 계층 |
| XSS | Jinja2 자동 이스케이프 유지, 사용자 입력(메모·여행명)은 저장 시 태그 제거, CSP 헤더 적용 | 템플릿·응답 헤더 |
| CSRF | Flask-WTF CSRF 토큰을 모든 상태 변경 요청에 요구 | 폼·AJAX 공통 |
| IDOR | 리소스 소유 검사 데코레이터 강제(11.1) | 라우트 계층 |
| 과도한 요청 | IP·사용자 기준 레이트 리밋(로그인 10/분, AI 추천 3/분) | Nginx + Flask-Limiter |
| 민감정보 노출 | 오류 응답에 스택트레이스 미포함, 상세는 서버 로그로만 | 전역 오류 핸들러 |

### 11.3 데이터 보호

* 전 구간 HTTPS(TLS 1.2+)를 강제하고 HTTP는 301 리다이렉트한다.
* 알레르기 정보는 건강 관련 민감정보로 분류하여 애플리케이션 레벨 AES-256 암호화 후 저장하고, 추천 프롬프트 외 용도로 사용하지 않는다.
* API 키·DB 자격증명은 환경변수로 주입하며 저장소에는 .env.example만 커밋한다(8.4절).
* 회원 탈퇴 시 소프트 삭제 후 30일 유예를 거쳐 배치 작업으로 개인정보를 파기·익명화한다(6장 데이터 수명 주기).

### 11.4 감사 로깅

* 로그인/로그아웃/실패, 권한 거부, 여행 삭제, 회원 탈퇴, Plan B 발동을 감사 이벤트로 기록한다(사용자 ID, IP, 시각, 결과).
* 로그에는 비밀번호·세션 토큰·알레르기 정보를 절대 기록하지 않는다.

### 12.3 데이터베이스 성능

* 6장의 인덱스 설계를 준수하고, 일정 조회는 (trip_id, day, order_no) 복합 인덱스로 커버링한다.
* N+1 방지를 위해 여행 상세 조회는 selectinload로 SCHEDULE·PLACE를 일괄 로딩한다.
* 슬로우 쿼리 로그(100ms 초과)를 활성화하여 주간 점검한다.

### 12.4 확장 경로

* 애플리케이션은 무상태(세션은 Redis 저장)로 설계하여 Gunicorn 워커 증설과 수평 확장이 가능하다.
* 경로 계산·AI 호출은 향후 Celery 작업 큐로 분리할 수 있도록 서비스 인터페이스를 동기/비동기 겸용으로 정의한다.
* 트래픽 증가 시 읽기 복제본(MySQL replica) 분리를 1차 확장안으로 한다.

## 17. 기대 효과 및 결론

### 17.1 기대 효과

* **사용자 관점:** 검색·지도·메모 앱을 오가며 수 시간 걸리던 여행 계획을 단일 서비스에서 수 분 내에 완성하고, 현장 변수(웨이팅·휴무·우천)에도 원터치로 대응할 수 있다.
* **기술 관점:** TSP 휴리스틱, LLM 후보-선별형 추천, 어댑터 기반 외부 연동·폴백 등 실무형 설계 패턴을 종합적으로 검증한다.
* **팀 역량 관점:** 요구사항 정의부터 설계·구현·테스트·배포·운영까지 전체 SDLC를 경험하며, 본 설계서 자체가 포트폴리오 산출물이 된다.

### 17.2 향후 발전 방향

* 모바일 앱(React Native) 확장과 오프라인 일정 열람.
* 실시간 협업 편집(WebSocket)과 투표 기반 일정 확정.
* 여행 후기·사진 아카이브와 추천 학습 고도화(수락/거절 피드백 반영 강화).
* 예약 연동(음식점·액티비티)과 지출 자동 정산(더치페이).

### 17.3 결론

RouteMate는 "장소 선택 → 최적 경로 → 상황 대응"이라는 여행 계획의 전 과정을 하나의 흐름으로 통합한다. 본 설계서는 요구사항 62건(FR 47, NFR 15), 설계 다이어그램 11종(UML 및 아키텍처·ERD), 테이블 10개, API 엔드포인트 40여 종, 화면 10개, 테스트 케이스 34건을 정의하여 구현 단계에서 참조 가능한 수준의 상세도를 확보하였다. 특히 세 핵심 엔진을 외부 의존성과 분리된 순수 모듈로 설계함으로써 테스트 용이성과 장애 내성을 동시에 달성하는 것이 본 설계의 중심 기여이다.
